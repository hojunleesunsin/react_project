P00015 useReducer 상세 가이드

대상 파일:
- page.tsx

========================================
1) 이 예제가 useReducer에 적합한 이유
========================================

P00015는 아래 상태들이 서로 연결되어 있다.

- query: keyword, team, status, sortBy, page, pageSize
- 서버 상태: isLoading, error, rows, totalCount, lastLoadedAt
- UI 보조 상태: selectedIds
- 동시성 제어 상태: requestId(오래된 응답 무시용)

이런 구조를 useState 여러 개로 나누면 다음 문제가 생긴다.

1. 상태 전이 규칙이 흩어진다.
   - 예: 키워드 변경 시 page를 1로 되돌리고, selection도 비워야 함.
   - 예: pageSize 변경 시 page=1 + selection 초기화.
   - 예: fetch 성공 시 selectedIds에서 현재 페이지에 없는 id 제거.
   -> 각 핸들러마다 setState를 여러 번 호출하며 중복 로직이 생긴다.

2. 의존 규칙 누락 위험이 커진다.
   - 한 핸들러에서는 page reset을 했는데 다른 핸들러에서는 빠뜨리는 식의 버그.

3. 요청 경합(race condition) 처리가 복잡해진다.
   - 느린 이전 요청이 나중에 도착해 최신 결과를 덮어쓰는 문제.
   - requestId 같은 조건 검사를 한 곳에서 일관되게 적용하기 어렵다.

useReducer는 "액션 -> 상태 전이"를 한 함수(reducer)로 모아서
규칙 일관성과 예측 가능성을 높여 준다.

========================================
2) reducer 구조 핵심
========================================

P00015 reducer는 크게 4개 도메인으로 액션을 나눴다.

1) query/*
- query/keyword, query/team, query/status, query/sort, query/page, query/pageSize, query/reset
- 필터 관련 액션마다 page reset, selection clear 같은 도메인 규칙을 같이 처리.

2) selection/*
- selection/toggle, selection/clear
- 선택 토글/초기화를 한 규칙으로 통합.

3) fetch/*
- fetch/start, fetch/success, fetch/error
- 로딩/에러/결과 반영을 일관되게 처리.

4) requestId 검사
- fetch/success, fetch/error에서 action.requestId !== state.requestId면 state 반환.
- 오래된 응답을 무시해 최신 요청만 화면에 반영.

========================================
3) 실행 흐름
========================================

[A. 사용자가 필터 변경]
1. dispatch({ type: "query/..." }) 호출
2. reducer가 query 갱신 + page 초기화 + selection 정리 등 규칙 반영
3. state.query 변경으로 useEffect 트리거
4. fetch/start -> fakeFetchMembers -> fetch/success or fetch/error 순으로 dispatch

[B. 조회 응답 도착]
1. reducer에서 requestId가 최신인지 확인
2. 최신이면 rows/totalCount/lastLoadedAt 반영
3. 오래된 요청이면 무시

[C. 체크박스 선택]
1. selection/toggle 액션으로 단일 규칙 적용
2. 조회 성공 시 현재 페이지에 없는 선택값 자동 정리

========================================
4) useState 대신 useReducer를 우선 고려할 기준
========================================

아래 조건 중 2~3개 이상이면 useReducer 검토 가치가 높다.

1. 상태가 4개 이상이고 서로 의존한다.
2. "A가 바뀌면 B/C도 함께 바꿔야 함" 같은 전이 규칙이 많다.
3. 비동기 요청 상태(loading/error/success)가 있다.
4. 액션 종류가 명확하다(추가, 수정, 필터변경, 초기화 등).
5. race condition 같은 방어 로직이 필요하다.
6. 팀 개발에서 상태 변경 추적/리뷰 용이성이 중요하다.

========================================
5) useReducer 장단점
========================================

[장점]
- 상태 전이 규칙을 중앙화
- 버그(누락/중복) 감소
- 액션 이름만 봐도 의도 파악 가능
- 테스트가 쉬움(reducer는 순수 함수)

[단점]
- 초기에 코드량 증가(action type, reducer 작성)
- 단순 화면에서는 과한 구조일 수 있음

결론:
- 단순 폼/단일 상태면 useState가 더 간단하다.
- P00015처럼 복합 상태 + 규칙 다수 + 비동기 경합이 있으면 useReducer가 더 안전하고 유지보수성이 높다.

========================================
6) 실무 팁
========================================

1. action 이름은 도메인/동사 형태로 통일
   - 예: "query/keyword", "fetch/success"

2. 파생값은 state에 저장하지 말고 계산
   - 예: totalPages는 useMemo로 계산

3. reducer 안에서 비동기 호출은 하지 않기
   - 비동기는 effect/핸들러에서 처리하고 결과만 dispatch

4. 상태 변경 이유를 액션으로 표현
   - 디버깅 시 "무슨 이유로 바뀌었는지"가 명확해진다.
